⚡ Why a Hybrid Approach?
A plain backtracking solver explores the search tree blindly — trying every digit, backing up only when a contradiction is hit. For an Expert puzzle with ~63 empty cells, this means up to 963 ≈ 1060 theoretical states. The hybrid solver slashes this to a few hundred recursive calls by combining four complementary techniques: AC-3 eliminates illegal values before search begins, MRV always picks the most constrained cell next, Forward Checking kills dead branches the instant they appear, and Dancing Links gives an ultra-efficient data structure for the underlying exact-cover search. Together they achieve speedups of 10,000–13,000× on Expert puzzles. Click any card below to read its full explanation.
01 — AC-3 (Arc Consistency 3)
Pre-search · O(e · d³)
▶

What it does: AC-3 propagates constraints before any search begins. It treats every pair of peer cells (same row, column, or 3×3 box) as a directed arc. For each arc (X → Y), it checks: does every value in X's domain have at least one compatible value in Y's domain? If not, those bad values are pruned from X's domain — and all arcs pointing back to X are re-queued for checking.

Sudoku specifics: A 9×9 grid has exactly 810 constraint arcs (each of 81 cells has 20 peers; 81 × 20 / 2 directed pairs × 2 directions = 810). Each cell starts with domain {1..9}; given cells have domain {their digit}. AC-3 often shrinks empty-cell domains from 9 candidates down to 2–3, and sometimes forces a unique value (when only one digit survives). On Easy puzzles AC-3 alone can solve the entire puzzle without any backtracking.

Failure detection: If any domain becomes empty during propagation, the puzzle has no solution and AC-3 returns a failure flag immediately — no backtracking search is ever started.

  • Initialise queue with all 810 directed arcs
  • Dequeue arc (Xi, Xj); for each value v in dom(Xi), check if any value in dom(Xj) is compatible
  • If no compatible value exists in dom(Xj), remove v from dom(Xi) (revise)
  • If dom(Xi) shrank, re-enqueue all arcs (Xk → Xi) for k ≠ j so they are rechecked
  • Repeat until queue is empty (success) or any domain hits size 0 (failure)
Variables: e = 810 arcs, d = 9 (max domain)
Time: O(e · d³) ≈ 590,000 ops worst-case
Space: O(e + n·d) — queue + domain storage
function AC3(domains): queue ← all 810 arcs (Xi, Xj) while queue not empty: (Xi, Xj) ← dequeue(queue) if REVISE(domains, Xi, Xj): if |domains[Xi]| = 0: return FAILURE for each Xk in peers(Xi) \ {Xj}: enqueue(queue, (Xk, Xi)) return SUCCESS
02 — MRV Heuristic (Minimum Remaining Values)
Variable ordering · O(n) per call
▶

What it does: MRV, also called the "fail-first" heuristic, decides which empty cell to fill next at every step of the backtracking search. Instead of filling cells left-to-right (top-left to bottom-right), MRV always picks the unassigned cell with the fewest remaining legal candidate values in its current domain.

Why fail-first works: A cell with only 1 remaining candidate has no branching at all — the assignment is forced, so it costs nothing extra. A cell with 2 values creates a binary branch instead of a 9-way branch. By tackling the most constrained cells first, the algorithm detects dead ends earlier and prunes whole subtrees before expanding them. This is the single biggest source of speedup over naive left-to-right selection.

Tie-breaking: When multiple cells share the minimum domain size, a secondary Degree heuristic (pick the cell with the most constraints on other unassigned cells) can be applied. This implementation uses first-found, which is sufficient for correctness and avoids the extra O(n) overhead of degree computation.

  • Scan all 81 cells; skip any that are already filled (board value ≠ 0)
  • Track the cell index with the smallest |domain| seen so far
  • Return that index — or −1 if all cells are filled, meaning the puzzle is solved
Time per call: O(n) = O(81) linear scan
Called: once per backtracking node visited
Effect: branching factor drops from 9 to ~1.5 average
function selectMRV(domains, board): best ← -1 bestSize ← 10 // sentinel > 9 for i in 0..80: if board[i] = 0 and |domains[i]| < bestSize: bestSize ← |domains[i]| best ← i return best // -1 → solved
03 — Forward Checking
Incremental pruning · O(20 · d) per assignment
▶

What it does: Whenever the search assigns digit v to a cell C, Forward Checking immediately removes v from the domains of all 20 peer cells (same row, column, and 3×3 box). If any peer's domain hits zero — meaning no legal digit remains for it — the assignment of v to C has created an impossible situation, so the solver backtracks at once without any further recursion.

How it differs from AC-3: AC-3 is a complete, global preprocessing pass run once before search — it enforces arc consistency across all 810 arcs and can propagate changes through many cells. Forward Checking is a lightweight, incremental cousin: it only checks the 20 arcs touching the newly assigned cell. It's faster per step (O(20) vs O(810)) but catches fewer conflicts. The two work together: AC-3 shrinks domains before search, Forward Checking maintains them during search.

Undoing on backtrack: Every pruned value is recorded in a list tied to the current assignment. When the solver backtracks and un-assigns C = v, every pruned value is restored to its peer's domain — keeping all domains perfectly consistent throughout the search tree.

  • On assign(C = v): iterate over all 20 peers P of cell C
  • If v is in domain[P], remove it and add P to the pruned list
  • If domain[P] becomes empty → restore all pruned values, return FAILURE
  • On backtrack: re-add v to domain[P] for every P in the pruned list
Time per assignment: O(20 · d) where d ≤ 9
Space: O(search depth × 20) for undo stack
Effect: eliminates dead branches before expansion
function forwardCheck(cell, val, dom): pruned ← [] for each peer P in peers(cell): if val ∈ dom[P]: dom[P].remove(val) pruned.append(P) if |dom[P]| = 0: restore(pruned, val, dom) return FAILURE return pruned // restore on undo
04 — Dancing Links / DLX (Knuth's Algorithm X)
Exact Cover · O(k) cover & uncover operations
▶

Exact Cover formulation: Sudoku maps perfectly onto the Exact Cover problem. A valid solution must satisfy all 324 constraints simultaneously: each of the 81 cells must contain exactly one digit (81 constraints), each digit 1–9 must appear exactly once in each row (81), column (81), and 3×3 box (81). Every possible placement — digit d in cell (r, c) — satisfies exactly 4 of these 324 constraints. DLX finds a set of 81 placements (rows in the matrix) that together cover all 324 constraints exactly once, with no overlap and no gap.

The Dancing Links data structure: The 729×324 binary constraint matrix is represented as a circular doubly-linked list of its non-zero (1) entries. Each node links left/right to other 1s in the same row and up/down to other 1s in the same column. A column header tracks how many 1s it currently contains. "Covering" a column removes it and all rows that intersect it from the active list in O(k) time, where k is the number of 1s in that column. "Uncovering" perfectly reverses this — the links were never destroyed, just bypassed — so backtracking is exact and O(k).

S-heuristic (MRV in disguise): At each DLX recursion level, the algorithm picks the column with the fewest 1s remaining. This is exactly MRV translated into the exact-cover representation: the column with the fewest 1s is the constraint that can be satisfied in the fewest ways — the most constrained choice. Choosing it minimises the branching factor at every step.

  • Build 729×324 sparse matrix; pre-cover columns for given-digit placements
  • If all columns covered → solution found, decode and return
  • Pick column C with fewest 1s (S-heuristic / MRV equivalent)
  • Cover column C; for each row R in C: select R, cover all columns R intersects, recurse
  • If recursion fails: uncover all columns R intersected, try next row; if no rows left, uncover C and return FAILURE
Matrix: 729 rows × 324 cols, ~2,916 ones
Cover/Uncover: O(k), k typically ≤ 9
Nodes explored: typically < 200 per puzzle
function DLX(header): if header.right = header: return SOLUTION FOUND C ← minOnes(header) // S-heuristic cover(C) for each row R in C.down..C: solution.push(R) for J in R.right..R: cover(J) if DLX(header): return TRUE solution.pop() for J in R.left..R: uncover(J) uncover(C) return FALSE
Full Hybrid Execution Flow
01Parse input. Accept 9×9 grid. Count empty cells. Validate no row, column, or box contains duplicate given digits before proceeding.
02Initialise domains. Given cells → domain = {their digit} (size 1). Empty cells → domain = {1, 2, …, 9} (size 9). Build peer lookup table: peers(i) = union of row ∪ column ∪ box cells, excluding i itself (always exactly 20 peers).
03Run AC-3. Enforce arc consistency across all 810 directed arcs. For each arc (Xi → Xj): prune values from dom(Xi) that have no compatible partner in dom(Xj). Re-enqueue affected arcs on any domain shrinkage.
04AC-3 result check. If any domain is now empty → puzzle is provably unsolvable, stop immediately. Otherwise, cells with singleton domains (forced values) are effectively pre-assigned — no branching needed for them.
05MRV cell selection. Find the empty cell with the smallest current domain. If no empty cells remain → all cells are filled, puzzle is solved — return SUCCESS.
06Candidate iteration. For the chosen cell, try each value v in its domain in order. Assign v, then run Forward Checking — remove v from all 20 peer domains, tracking every pruned entry.
07Recurse or backtrack. If Forward Checking found no empty peer domain → recurse to step 05. If recursion returns SUCCESS → propagate up. If FC failed or recursion failed → restore all pruned values, try next candidate v.
08Exhausted all candidates. If every value in the cell's domain has been tried and failed → return FAILURE to the parent call, triggering backtrack one level up.
09Record metrics. Stop the wall-clock timer. Store recursive call count, AC-3 arc checks, and elapsed time. Update the Live Comparison panel and push a live data point to the Performance Graph.
